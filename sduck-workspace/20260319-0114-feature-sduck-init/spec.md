## 1. 배경 및 목적

### 문제 / 요구사항

현재 프로젝트에는 엄격한 TypeScript/ESLint/Husky 기반은 갖춰졌지만, 실제 SDD 워크플로우를 시작하는
첫 명령인 `sduck init`은 아직 구현되지 않았다.

`sduck init`은 사용자가 어떤 저장소에서든 SDD 작업을 시작할 수 있도록 초기 디렉토리 구조와 타입 정의,
기본 asset을 생성하는 부트스트랩 명령이어야 한다.

또한 이미 일부 설정이 존재하는 저장소, 사용자 커스텀 파일이 섞여 있는 저장소, 일부 파일만 누락된 저장소처럼
실제로 자주 발생할 수 있는 엣지 케이스를 안전하게 처리할 수 있어야 한다.

### 기대 효과

- 사용자가 한 번의 명령으로 `.sduck/`, `sduck-workspace/`를 초기화할 수 있다
- 이미 설정이 존재하는 경우, 기본 동작에서 기존 파일을 지우지 않고 유지 정책을 명확히 안내할 수 있다
- 기본 타입 정의와 스펙 템플릿이 일관된 형태로 생성된다
- 이후 `sduck start` 등 후속 명령이 기대하는 파일 구조가 안정적으로 보장된다

- [x] AC3: 기본 asset 파일이 필요한 위치에 생성되거나, 이미 존재하면 안전하게 유지된다
- [x] AC4: 이미 초기화된 프로젝트에서 재실행 시 덮어쓰기/스킵/질의 정책이 일관되게 동작한다
- [x] AC5: 기존 설정 또는 사용자 커스텀 파일이 있을 때 무단 덮어쓰기 없이 안전하게 동작한다
- [x] AC6: `sduck init --force` 옵션으로 기본 asset과 타입 정의를 강제 재생성할 수 있다
- [x] AC7: 실제 파일 시스템 기준 e2e 테스트로 신규 초기화, 재실행, 충돌 상황을 검증한다

### 기능 상세 설명

- 명령어: `sduck init`
- 생성 대상 기본 구조:

```text
프로젝트 루트/
├── sduck-assets/
│   ├── plan-evaluation.yml
│   ├── spec-build.md
│   ├── spec-feature.md
│   ├── spec-fix.md
│   ├── spec-refactor.md
│   └── spec-chore.md
└── sduck-workspace/
```

- 기본 명령 `sduck init`은 누락된 디렉토리/파일만 생성하는 방식이어야 한다
- 이미 존재하는 사용자 파일을 무단으로 덮어쓰지 않아야 한다
- 이미 설정이 있는 상태에서 충돌 가능 항목이 있으면, 기본 동작에서는 유지하고 사용자에게 유지/스킵 사실을 출력한다
- 강제 재생성이 필요한 경우 `sduck init --force`로 기본 asset을 다시 생성할 수 있어야 한다
- 어떤 파일이 생성되었고 어떤 파일이 유지되었고 어떤 파일이 강제 갱신되었는지 CLI 출력으로 구분해 보여줄 수 있어야 한다
- `sduck-assets/`의 템플릿은 이후 `sduck start`가 직접 참조할 수 있는 최소 구조를 포함해야 한다

### 엣지 케이스

- 프로젝트 루트에 `sduck-workspace/`만 있고 `.sduck/`가 없는 경우
- `sduck-assets/`는 있으나 템플릿 일부만 누락된 경우
- `sduck-assets/plan-evaluation.yml`만 누락된 경우
- 기존 템플릿 파일이 사용자에 의해 수정된 경우
- 동일 명령을 여러 번 반복 실행하는 경우
- 파일/디렉토리 생성 권한이 없는 경우
- 파일 대신 동일 이름의 디렉토리 또는 디렉토리 대신 파일이 이미 존재하는 경우

## 4. UI/UX 명세 (해당 시)

CLI 출력 기준으로 정의한다.

### 인터랙션 정의

- 성공 시 생성된 항목과 유지된 항목, 강제 갱신된 항목을 구분해 출력한다
- 이미 초기화된 경우 기본 동작은 noop에 가까운 성공으로 처리하되, 변경 없음 또는 유지된 항목을 명확히 안내한다
- 충돌 가능 항목이 있더라도 기본 동작에서는 지우지 않고 유지한다
- `--force` 사용 시에만 기본 템플릿/타입 파일 재생성을 허용한다
- 에러 메시지는 누락 경로, 권한 문제, 파일 충돌 여부를 알 수 있게 작성한다
