## 1. 배경 및 목적

### 문제 / 요구사항

현재 테스트는 대부분 임시 디렉토리나 개별 fixture 기반으로 동작한다. 하지만 사용자는 프로젝트 루트 안에 실제 테스트용 작업 디렉토리를 두고, 테스트 시작 시 내용을 비운 뒤 `sduck` 명령을 직접 실행하면서 실제 파일이 생성되는지 검증하는 흐름을 원한다. 이때 전역 설치된 `sduck`와 혼동되지 않도록, 테스트는 반드시 현재 저장소에서 개발 중인 로컬 CLI만 실행해야 한다.

### 기대 효과

- 프로젝트 내부의 고정된 테스트 작업 디렉토리에서 실제 CLI 사용 흐름을 더 가깝게 검증할 수 있다.
- 테스트 시작 전 디렉토리 초기화와 파일 생성 검증 규칙이 명확해진다.
- 글로벌 설치본이 아니라 현재 저장소의 로컬 CLI를 사용한다는 보장이 생긴다.
- init/start/spec approve/plan approve 같은 명령의 실제 파일 생성 결과를 확인하기 쉬워진다.

## 2. 기능 명세

### 사용자 스토리

```text
As a CLI maintainer,
I want tests to run inside a real project test workspace that is cleaned before each run,
So that I can verify actual command execution and generated files more directly.
```

### 수용 기준 (Acceptance Criteria)

> 모든 항목이 충족되어야 이 작업이 완료(Done)로 간주된다.

- [ ] AC1: 프로젝트 내부에 고정된 테스트용 작업 디렉토리 구조 또는 이를 관리하는 공용 helper가 추가된다.
- [ ] AC2: 관련 테스트는 시작 전에 해당 디렉토리를 비우거나 초기화한 뒤 현재 저장소의 로컬 `sduck` CLI를 실행한다.
- [ ] AC3: 테스트는 stdout만이 아니라 실제 생성된 파일/디렉토리 결과를 검증한다.
- [ ] AC4: 기존 테스트 간 간섭 없이 반복 실행 가능해야 한다.

### 기능 상세 설명

- 테스트용 작업 디렉토리는 프로젝트 루트 안의 명확한 경로에 두되, 실수로 일반 소스와 섞이지 않아야 한다.
- 테스트 시작 시 디렉토리를 비우는 동작은 예측 가능하고 반복 가능해야 한다.
- CLI 실행 경로는 글로벌 설치본이 아니라 현재 저장소의 `src/cli.ts` 또는 빌드 산출물처럼 로컬 엔트리만 사용해야 한다.
- 실제 명령 실행 결과로 생성된 `sduck-assets/`, `sduck-workspace/`, rule file, task files 등을 검증할 수 있어야 한다.
- 가능하면 여러 e2e 테스트가 공통 helper를 통해 같은 패턴으로 테스트 작업 디렉토리를 준비해야 한다.

## 5. 테스트 계획

### 단위 테스트 (Unit Test)

| task flow in real test workspace | 로컬 `start`/`approve` 실행               | workspace/task file 생성 및 상태 변경 검증 |

## 6. 영향 범위 분석

### 변경 파일 목록 (예상)

```text
target_paths:
  - test/...
  - tests/helpers/...
  - tests/e2e/...
  - 필요 시 CLAUDE.md
  - 필요 시 AGENT.md
```

### 사이드 이펙트 검토

- 테스트가 고정 디렉토리를 공유하면 병렬 실행 간 충돌 가능성이 있다.
- 디렉토리 비우기 로직이 잘못되면 의도치 않은 파일 삭제 위험이 있다.
- 로컬 CLI 대신 전역 `sduck`가 실행되면 테스트 결과가 현재 브랜치 코드와 어긋날 수 있다.

- 영향 가능성 있는 모듈: e2e helper, 파일 정리 로직, CLI 실행 경로, CI 테스트 실행 방식
- 회귀 테스트 필요 영역: 모든 e2e 명령 실행, 테스트 격리성

### 롤백 계획

- 고정 테스트 작업 디렉토리 전략을 되돌리고 기존 temp workspace 방식으로 복귀한다.
